# Ontology principles

## Vendor-neutral by default

Objects should describe roles, responsibilities, interfaces, and controls without assuming a specific AI provider, orchestrator, cloud, or policy engine.

## Security-conscious design

Schemas should encourage explicit trust boundaries, sensitivity labeling, least-privilege permissions, and auditability. Sensitive values should be referenced or redacted rather than stored inline.

## Governance-first structure

Districts, policies, permissions, workflows, and events should make control points visible so implementations can map them to review, approval, and enforcement systems.

## Composable objects

Core entities should be independently reusable and linked through stable identifiers rather than hidden coupling.

## Portable semantics

Names and fields should favor broad interoperability, clear definitions, and conservative assumptions about execution environments.

## Extensible without fragmentation

Schemas may allow controlled extension points, but the base vocabulary should remain stable and easy to validate.
